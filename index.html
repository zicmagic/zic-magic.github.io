内存优化

Posted on 2018-10-25

本文主要介绍一下常见的内存问题及处理办法

#野指针

遇到EXC_BAD_ACCESS的问题，可能是野指针问题。但是此处控制台不会打印更多的信息，无法准的定位问题，可以点击选择 Product - Scheme -Edit Scheme - Run - Diagnostics - Zombie Objects ，这样控制台会打印出一些原因和内存地址，然后查看控制左侧的对象信息,通过内存地址定位到确定的对象。如图所示是dataScource的问题,查看属性定义发现是错误地使用了assign。

这里讲一下，assign和weak的区别：

weak 声明的变量对象释放后会自动清空，赋值为nil。

assign 声明的变量对象释放后不会自动赋值为nil。

#内存泄漏

这里介绍一下检测内存泄漏的一般方法：

1、静态检测方法

2、动态检测方法 instruments 和第三方内存检测工具（暂时先不介绍，后续再介绍）

3、析构方法 dealloc

##静态检测方法

遇到EXC_BAD_ACCESS的问题，经过上一段落的处理办法也无法定位的，因为僵尸对象的检测，只能针对已被释放的对象的地址，无法定位非僵尸对象的问题，就需要使用静态分析的办法，。静态分析的办法分为手动和自动两种，分开介绍。

###手动静态检测

点击Product - Analyze 可以开启静态检测。

如图所示，可以看出是指针为null的问题。但是循环引用的问题的在静态检测中还是无法识别出来。

###自动静态检测

自动开启静态检测的话如下图所示，在编译的时候会自动进行检测，我们平常写代码的时候也可以使用，但是工程较大的时候会影响编译速度，不建议直接开启，可以在后期进行内存优化的时候打开。

##动态检测方法 instruments

打开静态分析的工具界面的方法很多这里就不介绍了。

轴线图上有红色叉叉的地方代表检测到内存泄漏，右边画红色圈圈的地方发开能直接跳转定位到代码的界面，也十分的简单，但是也依然会有一些错误无法定位，比如block的循环引用。

##析构方法

当某个对象的引用计数为0的时候，会自动调用 dealloc 的方法。

一般block的循环引用的问题，可以用此方法进行检测。解决的办法就是用 __weak typeof (self) weakSelf = self 用 weakSelf 代替 self 就可以了。

OC必须知道的小知识

Posted on 2018-07-05

#必须知道的小知识

###问题1 Category(类别)、 Extension(扩展)和继承的区别

1、分类有名字，类扩展没有分类名字，是⼀一种特殊的分类。
2、分类只能扩展⽅方法(属性仅仅是声明，并没真正实现)，类扩展可以扩展属性、成员变量和⽅法。
3、 继承可以增加，修改或者删除⽅方法，并且可以增加属性。

###问题2 MVC模式中包含的常用设计模式
代理模式、命令模式（action）、观察者模式、中介者模式（mvc本身）、策略模式（view与model的
交互）

###问题3 oc中字典实现的原理
1、底层是一个哈希表（数组）
2、根据哈希函数（算法算法 md5 SHA1）找到相应的位置，时间复杂度为1，哈希函数是压缩的印象
3、地址重复的时候采用开放地址法，给一个增量。时间复杂度为x

###问题4 静态库和动态库的区别

静态库：链接时会被完整的肤质到可执行的文件之中，被多次使用就有多分拷贝
动态库：链接时不复制，程序运行时由系统动态加载到内存，系统只加载一次，多个程序共用、节省内存

Hook说明：

1、所有的 UIControl 类型的控件、UITabBarButton 以及在导航栏上自定义添加的 UIBarButtonItem 的点击事件，都可以通过 hook 系统类 UIApplication 的 sendAction:to:from:forEvent: 方法进行拦截。但是，这个方法并不能拦截到导航栏上系统自动添加的返回按钮的点击，实际中没有用到系统导航栏的话就可以不hook，如果需要就可以通过navigationBar:shouldPopItem: 方法来实现对它的点击的拦截。

2、针对与手势相关的事件，首先通过 hook 系统类 UIGestureRecognizer 的 initWithTarget:action: 与 addTarget:action: 这 2 个方法拿到 target 对象与 action 方法，然后再去 hook target 的 action 方法，从而能够拦截到手势相关的事件，但是本方案中的点击事件的记录不是在此方案中记录的，这里只是用来获取手势的方法名。具体的作用后面进行说明。而是通过hook 系统类 UIView 的 addGestureRecognizer，这样可以方便的获取类名用于行成路径。

3、对于 UITableView、UICollectionView 某一行的点击，首先 hook 它们的 setDelegate: 方法，从而拿到 delegate 对象，然后再去 hook delegate 的 didSelectRowAtIndexPath: 方法即可。

4、对于启动和退出app的事件，则需要通过hook 系统类 NSNotificationCenter 的 postNotification 和 postNotificationName:object:userInfo: 通过捕捉 UIApplicationDidBecomeActiveNotification 、 UIApplicationWillResignActiveNotification 和 UIApplicationWillTerminateNotification 分别记录此类事件。另外如果需要记录输入框的事件也可以在此处拦截 UITextFieldTextDidChangeNotification

5、对于UIViewController的进出事件 ，有一些其他方案则是通过 viewWillAppear 进行统计记录，个人认为这里使用 viewDidAppear 会合理一点。

6、针对滑动事件的处理目前还没有集成，目前需求还没有到此处。

##Event Collector

这些要收集的事件数据中包含一些基本信息，分为通用信息和事件信息，通用信息统一放在接口中传输，通用信息包括： userid、 productName（bundleID用于识别不同的项目） 、 version、 osType、osVersion 、 osDeviceType 、networkType 、 isp 、clientid（这里采用IDFA和SIDFA结合）、 gps（如果应用中开启过GPS功能和权限会自动获取否则不获取）。除此之外，还有一些与特定事件相关的信息，例如对于 view 的点击事件，还需要收集与 view 的相关信息；对于列表行的点击，还需要收集点击行的 indexPath 信息。事件信息包括： type 、 uniqueId、 className 、 createTime、 page 、 path 、 functionDescription（功能描述参数） 、 otherDescription（目前没有使用、预留传参参数）

下面针对一些特别的参数做一些描述

###type

###path

下面说明这个参数如何获取的。先关键附上代码

整个 APP 的视图结构可以看成是一颗树（viewTree），树的根节点就是 UIWindow，树的枝干由UIViewController及UIView组成，树的叶节点都是由UIView组成。

那么在viewTree中用什么信息来表示其中任意一个 view 的位置呢？很容易想到的就是使用目标 view 到根之间的每个节点的深度（层次）组成一个路径，而节点的深度（层次）是指此节点在父节点中的 index。这样确实能够唯一的表示此 view 了，但是有一个缺点：它的可读性很差。因此在此基础上又增加了每个节点的名称，节点的名称由当前节点的 view 的类名来表示。同时在开头都添加了一个页面名称作为标识。

因此，在 viewTree 中，由一个 view 到根节点之间的每个节点的名称与深度（层次）共同组成的信息构成了此 view 的viewPath。另外，由于在做 view 的统计分析时，都是以页面为单位的，因此 SDK 在生成 viewPath 时，只到 view 所在的 UIViewController 级别，而非根部的 UIWindow。这样做也在一定程度上减少了viewPath 的长度。

项目中 UITableView 和 UICollectionView 的树级关系没有到每个具体的 cell ，避免产生很多无用的id，而是将 indexpath 作为描述信息传入。上述已经描述过了。

GCD的使用和API介绍

名次解释

线程：程序中执行任务的最小单位

任务：简单的来说就是代码段，在GCD中可以理解为block的内容。

队列：用存放任务的数组

异步执行：任务创建后可以绕过当先线程的任务，开辟一条新的线程去执行任务。异步执行则不可

并行队列： 队列中的任务可以同时执行。串行队列则要按顺序执行。

串行队列嵌套

串行队列需要等block1，走完才会走block2

串行队列中不能使用两个同步互相嵌套，两个同步的嵌套会造成两个任务的相互等待，会造成死锁

主队列

主队是一个串行队列，拥有串行队列的特点，但是异步执行的时候不会开辟子线程，需要等待主线程中的任务执行完成才会执行，同理主队列中也不能添加同步任务

GCD常用API

apply

添加多个异步任务,类似for循环添加一步任务，但是有细微区别，end会最后执行

如果block中是一步任务则不会阻塞当前线程，这个可以自己尝试一下，end会提前打印。

但是任务中再添加异步，通知会失效，需要使用enter和leave的功能,每个任务中都要使用。apply中则没有这样的功能。这个功能可以用在异步下载最后合成的功能中。

##barrier

barrier起来的是栅栏的作用，可以将1234分段执行

semaphore

使用信号量的个数只会有相应个数的线程在运行，即并发数

当信号量为1的时候就能保证线程的安全，即只有一条线程对当前对象进行操作

实际上semaphore就是一种线程锁下面再介绍两种常用的线程锁

synchronized

synchronized在县城锁当中的性能是最差的，在数量级大的情况下不建议使用，会造成严重的卡顿

nonatomic和atomic

nonatomic和atomic,字面意思就是原子性和非原子性，atomic会在set/get的时候加锁，但是会影响使用的性能。

从打印结果来看并没有得到预期的10000，因为只能在set/get的时候使用，使用上非常的有局限性并不能保证线程的安全而且十分的影响性能。

可以使用NSLock来解决刚刚出现的问题
